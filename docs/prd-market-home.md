## 1. 背景

当前 App 已经具备自选、持仓、触发和盯盘等与个股观察相关的功能，但用户在进入 App 后，还缺少一个能够快速回答以下问题的行情首页：

- 今天整体市场是上涨还是下跌？
- 哪些行业或主题正在变热？
- 市场成交和资金活跃度如何？
- 有哪些重要资讯或盘中异动需要关注？
- 我应该继续查看自选、盯盘，还是进入某个行业？

本需求新增或整理一个面向盘中快速浏览的“行情首页”。页面不直接替代专业行情终端，也不在 MVP 阶段承载完整交易、复杂选股或深度投研功能，而是作为用户进入行情模块后的第一层信息入口。

## 2. 行业调研结论

### 2.1 主流行情产品的共同模块

从同花顺、东方财富、指南针、Wind 等行情产品的公开功能说明和行情数据页面看，证券行情产品通常会同时展示以下信息：

1. **核心指数**：上证、深证、创业板等主要指数的最新点位、涨跌额和涨跌幅。
2. **市场宽度**：上涨、下跌、平盘家数，以及涨停、跌停等极端表现数量，用于快速判断市场情绪。
3. **成交和资金**：两市成交额、主力资金或行业资金流向，用于判断市场活跃度和资金偏好。
4. **行业/板块**：行业指数涨跌幅、资金净流入、行业内领涨股、成分股数量等。
5. **概念/主题**：热点概念、事件驱动、概念涨跌幅、相关龙头或代表股票。
6. **实时资讯和异动**：实时快讯、行业消息、指数异动、个股异动和公告入口。
7. **排行和快捷入口**：涨幅榜、跌幅榜、资金榜、自选、盯盘、选股等入口。

### 2.2 本项目的取舍

行业功能很多，但移动端行情首页不能把所有数据平铺。结合当前 App 的视觉语言和本次还原度测试目标，本期采用以下优先级：

- 第一优先级：指数和市场整体状态。
- 第二优先级：行业热点和资金方向。
- 第三优先级：实时简报和下一步操作入口。
- 暂不在首屏完整展示：复杂 K 线、盘口五档、完整行业树、策略选股、龙虎榜明细。

## 3. 目标

让用户在打开行情首页后的 10 秒内完成市场扫描，知道当前市场状态、主要热点和下一步入口，并能够进入行业详情、自选或盯盘继续查看。

## 4. 问题定义

当前如果只提供单个指数或股票列表，用户需要在多个页面之间切换，才能判断市场整体状态和行业方向。页面也容易退化为通用金融卡片堆叠，缺少证券 App 常见的市场层级和盘中扫描节奏。

本需求需要解决：

- 核心指数和市场宽度没有被组织成一个明确的市场摘要。
- 行业热点与指数行情之间缺少关联，用户看不到“哪里在涨、谁在带动、资金是否进入”。
- 实时资讯和盘中热点没有稳定的展示位置。
- 页面缺少通向自选、盯盘和行业详情的明确入口。
- 原型生成时容易只生成静态卡片，无法演示基本页面信息架构。

## 5. 目标用户

### 5.1 主要用户

已有股票关注列表、在交易时段内需要快速浏览大盘和行业热点的普通投资者。

### 5.2 次要用户

- 已持仓、需要判断市场环境再做观察或操作的用户。
- 暂未确定交易标的、希望先了解热门行业和市场情绪的用户。
- 需要在需求评审中查看行情首页结构和交互路径的产品、设计和研发人员。

## 6. 用户场景

### 6.1 开盘后快速看盘

用户在开盘后进入 App，先查看主要指数、上涨下跌家数和成交额，再判断今天是普涨、分化还是弱势行情。

### 6.2 寻找行业热点

用户看到某个行业涨幅靠前，想知道行业当前涨幅、资金净流入、领涨股和行业内股票数量，并进一步进入行业详情。

### 6.3 盘中跟踪变化

用户在盘中再次打开行情首页，希望看到数据更新时间、市场变化和最新简报，而不是上一交易日的静态内容。

### 6.4 从行情进入个人任务

用户发现某个行业或个股值得继续观察，需要进入自选、设置盯盘，或查看已经存在的触发提醒。

## 7. 范围

### 7.1 范围内

- 新增或整理移动端行情首页。
- 默认展示 A 股市场行情。
- 支持市场范围切换的交互演示：A 股、港股、美股；非 A 股可使用演示数据。
- 展示核心指数摘要。
- 展示市场涨跌结构和成交额摘要。
- 展示行业热点列表。
- 展示行业涨跌幅、资金方向、领涨股和成分股数量等核心字段。
- 展示实时简报或盘中资讯列表。
- 提供自选、盯盘、行业详情和排行入口。
- 支持首页到行业详情、实时简报详情、自选和盯盘的基本跳转演示。
- 支持加载、数据更新时间、数据异常和非交易时段状态。
- 默认生成亮色 390px HTML 可交互原型。

### 7.2 可选范围

- 指数卡片横向切换。
- 行业列表按涨幅、资金净流入切换排序。
- 行情首页下拉刷新反馈。
- 行业热点的“加入自选”或“设置盯盘”入口。
- 最近浏览行业或个股的轻量入口。

### 7.3 范围外

- 真实交易、下单、撤单和交易确认。
- 完整的个股详情页、K 线和五档盘口。
- 完整行业树、概念树和多级筛选器。
- 复杂的智能选股或策略回测。
- 龙虎榜、融资融券、北向资金等专业数据的完整详情。
- 新闻资讯全文阅读和内容生产系统。
- 实时行情数据接口、数据供应商接入和正式数据授权。
- 个性化推荐模型和投资顾问结论。
- 桌面端行情工作台。

## 8. 页面目标与信息架构

### 8.1 页面目标

行情首页只承担三件事：

1. 解释当前市场状态。
2. 帮助用户发现行业和热点方向。
3. 把用户送往下一步页面或个人任务。

### 8.2 页面层级

```text
行情首页
├── 市场范围切换
├── 市场摘要
│   ├── 核心指数
│   ├── 上涨/平盘/下跌家数
│   └── 成交额与更新时间
├── 实时简报
├── 行业热点
│   ├── 涨幅榜
│   ├── 资金榜
│   └── 行业详情
├── 快捷入口
│   ├── 自选
│   ├── 盯盘
│   ├── 全部排行
│   └── 资金流向
└── 底部导航
```

### 8.3 原型必须演示的主路径

```text
行情首页
  → 点击行业热点
  → 行业详情页
  → 点击返回
  → 回到行情首页

行情首页
  → 点击实时简报
  → 简报详情页或详情弹层
  → 关闭/返回
  → 回到行情首页

行情首页
  → 点击“自选”或“盯盘”快捷入口
  → 进入对应业务页签
```

页面原型不要求实现完整行业详情数据，但必须让用户看出入口、页面层级、返回关系和主要任务闭环。

## 9. 功能需求

### F-01 市场范围切换

#### 用户意图

用户希望查看不同市场的行情首页。

#### 展示内容

- `A股`
- `港股`
- `美股`

#### 系统行为

- 默认选中 `A股`。
- 用户点击市场范围后，指数名称、行情数据、行业/板块文案和更新时间同步切换。
- MVP 原型可以使用静态演示数据，但必须明确数据为演示数据。

#### 规则

- 市场范围切换使用短 Tab 或 Segment，不使用下拉菜单替代主入口。
- 不同市场的数据单位、涨跌颜色和交易时间说明不能混淆。

### F-02 市场摘要

#### 用户意图

用户希望第一眼知道市场整体表现。

#### 必须展示

- 当前市场名称，例如 `A股市场`。
- 更新时间，例如 `数据更新于 10:58`。
- 上证指数：最新点位、涨跌额、涨跌幅。
- 深证成指：最新点位、涨跌额、涨跌幅。
- 创业板指：最新点位、涨跌额、涨跌幅。
- 两市成交额。
- 上涨家数、平盘家数、下跌家数。

#### 可选展示

- 涨停家数、跌停家数。
- 市场情绪或活跃度短标签。
- 指数分时走势小图。

#### 交互

- 点击指数卡片进入指数详情页或展示详情入口反馈。
- 点击成交额或涨跌结构进入市场概览详情。
- 点击刷新后展示更新时间变化或加载状态。

#### 数据规则

- 上涨/正向数据使用项目定义的金融正向颜色。
- 下跌/负向数据使用项目定义的金融负向颜色。
- 页面不得只使用颜色表达涨跌，必须同时展示数值或文字。

### F-03 市场涨跌结构

#### 用户意图

用户希望知道当前行情是普涨、普跌还是结构性分化。

#### 必须展示

- 上涨家数。
- 平盘家数。
- 下跌家数。
- 至少一个情绪解释，例如：
  - `上涨占优`
  - `涨跌分化`
  - `下跌占优`

#### 可选展示

- 涨停/跌停数量。
- 市场宽度横向比例条。
- 量能相较昨日的变化。

#### 规则

- 情绪文案必须根据演示数据保持一致，不得出现“上涨家数较多”但页面数字显示下跌占优的矛盾。
- 不在 MVP 中生成投资建议，例如“适合追涨”“建议买入”。

### F-04 实时简报

#### 用户意图

用户希望快速了解影响市场或行业的最新信息。

#### 必须展示

- 简报标题。
- 来源或类型标签，例如 `市场`、`行业`、`公告`。
- 发布时间。
- 2-4 条简短内容。
- `更多` 或可点击的详情入口。

#### 系统行为

- 默认按时间倒序展示。
- 点击简报进入简报详情页或弹层。
- 未登录用户可以查看公开简报摘要；是否允许查看全文由后续权限策略确定。

#### 规则

- 标题必须是具体事件，不使用“市场最新动态”这类空泛标题。
- 不在首页展示长段全文。
- AI 生成内容必须带有来源或“AI 整理”标识，并显示风险提示。

### F-05 行业热点

#### 用户意图

用户希望知道当前哪些行业值得进一步查看，以及行业上涨是否有资金和个股带动。

#### 必须展示

每个行业条目至少展示：

- 行业名称。
- 行业涨跌幅。
- 领涨股名称。
- 领涨股涨跌幅。
- 行业成分股数量或参与股票数。

建议展示：

- 主力资金净流入/净流出。
- 行业指数点位。
- 行业状态标签，如 `热点`、`资金流入`、`异动`。

#### 默认排序

- 默认按行业涨跌幅降序。
- 提供切换到“资金净流入”排序的入口。
- 排序按钮只能改变排序字段，不改变行业条目的基本结构。

#### 交互

- 点击行业条目进入行业详情页。
- 点击领涨股进入个股详情或展示跳转反馈。
- 点击排序切换按钮后，列表顺序和排序标签同步更新。
- 点击行业操作入口可选择加入关注或设置盯盘；本期可以只展示入口和结果反馈。

#### 行业详情页最低信息要求

原型中如果演示行业详情页，至少包含：

- 行业名称和当前涨跌幅。
- 行业指数或趋势摘要。
- 领涨股和代表股票列表。
- 行业资金方向。
- 相关实时资讯。
- 返回行情首页入口。

### F-06 快捷入口

#### 用户意图

用户从行情首页快速进入个人关注或专业数据功能。

#### 必须展示

- `自选`
- `盯盘`
- `涨跌排行`
- `资金流向`

#### 交互

- 点击 `自选` 进入当前 App 自选页。
- 点击 `盯盘` 进入盯盘页签。
- 点击 `涨跌排行` 进入排行列表或展示页面入口。
- 点击 `资金流向` 进入资金流向列表或展示页面入口。

#### 规则

- 快捷入口使用当前 App 的 Quick Action/Grid 视觉模式。
- 不能因为行业功能较多而扩展成 3×4 或更大的功能宫格。
- 首屏只展示高频入口，其余功能进入“更多”或后续页面。

### F-07 底部导航

#### 系统行为

- 保持当前 App 的底部导航结构和视觉样式。
- 当前页面对应的行情入口处于选中态。
- 点击自选、交易等入口时进入对应业务页面或原型占位页面。

#### 规则

- 不在本需求中新增底部导航 Tab。
- 不将“行业”“排行”“资金流向”等二级能力直接加入底部导航。

### F-08 数据刷新和更新时间

#### 系统行为

- 页面展示最近更新时间。
- 用户下拉或点击刷新后展示加载状态。
- 刷新失败时保留上一次数据，并展示“更新失败，请稍后重试”。
- 数据超过定义的时效阈值时，展示“数据可能存在延迟”。

#### 原型要求

- 至少演示一次刷新反馈。
- 至少演示一次数据加载或异常状态。

### F-09 非交易时段

#### 系统行为

- 非交易时段显示“已收盘”或“盘前/盘后”状态。
- 保留上一交易日收盘数据，并明确数据日期。
- 实时简报仍可展示，但不伪装成实时行情。

## 10. 页面与视觉约束

### 10.1 来源优先级

1. Figma Page 1 中的真实行情首页结构和组件视觉。
2. 当前项目已确认的亮色 token 和明暗色映射。
3. `stock-trading-prototype-design-system` 的页面模式和组件规则。
4. 本 PRD 中的行业功能要求。
5. 若源稿没有明确证据，必须将新增模块标记为“基于行业惯例的原型假设”。

### 10.2 页面约束

- 默认输出 390px 移动端 HTML 原型。
- 默认亮色模式。
- 优先使用当前 App 的浅色背景、蓝色高亮、紧凑列表、轻量卡片和底部导航。
- 不做通用金融 SaaS 仪表盘。
- 不使用营销 Hero、装饰性渐变或大面积宣传文案。
- 不把所有行业字段堆在一张大卡片中。
- 内容、间距、圆角、文字层级和图标应尽量使用当前 App 已确认规则。
- 通用功能图标优先调用项目图标 Sprite；特殊行业图标可以使用明确标注的临时图标。

### 10.3 信息密度

- 首屏必须能看到市场范围、主要指数和至少一个下一步入口。
- 行业列表优先使用可扫描的列表行，不使用过大的行业卡片。
- 简报内容保持 2-4 行摘要，避免长文章挤占行情内容。
- 核心数据不依赖 hover、长按或隐藏手势才能发现。

| source | string | 行情快讯 | 来源或来源类型 |
| publishedAt | datetime | 10:46 | 发布时间 |
| summary | string | 板块内多只个股涨幅扩大 | 首页摘要 |
| isAiGenerated | boolean | false | 是否由 AI 整理 |

## 12. 状态要求

### 12.1 首次加载

- 展示指数、市场宽度、行业列表和简报的骨架占位。
- 不展示错误的 0 值作为真实数据。

### 12.2 正常状态

- 所有核心模块展示更新时间。
- 数据颜色、单位、精度和排序一致。

### 12.3 空状态

- 行业数据为空时展示“暂无行业数据”，并保留返回或刷新入口。
- 简报为空时展示“暂无最新简报”，不显示空白卡片。

### 12.4 异常状态

- 行情服务异常时，显示“行情暂不可用”。
- 行业数据异常时，不影响指数和自选入口展示。
- 单个模块失败时采用局部错误，不阻塞整个行情首页。

### 12.5 非交易时段

- 明确显示市场状态和数据日期。
- 禁止使用“实时”字样描述上一交易日数据。

## 13. 权限、合规和风险提示

- 未登录状态可查看公开行情摘要，是否允许查看完整数据由产品和数据授权确认。
- 所有行情数据需要显示来源、时间或延迟标识。
- AI 整理的简报必须与人工资讯或原始来源区分。
- 页面不输出个性化买卖建议，不使用“建议买入”“适合追涨”等表达。
- 页面底部或相关模块保留“市场有风险，投资需谨慎”等风险提示，具体文案由合规确认。
- 原型中的数字为演示数据，不能让评审误认为接入真实行情。

## 15. 验收标准

### 15.1 页面结构

- [ ] 用户进入页面后默认看到 `A股` 行情首页。
- [ ] 首屏同时出现市场范围、核心指数和至少一个快捷入口。
- [ ] 页面保留当前 App 底部导航，不新增底部 Tab。
- [ ] 页面模块顺序符合“市场摘要 → 简报/热点 → 快捷入口”的信息层级。
- [ ] 页面不是通用金融仪表盘或营销落地页。

### 15.2 指数和市场状态

- [ ] 页面展示上证指数、深证成指、创业板指的名称、点位和涨跌信息。
- [ ] 页面展示上涨、平盘、下跌家数。
- [ ] 页面展示成交额和数据更新时间。
- [ ] 上涨、下跌、平盘的数值和颜色语义一致。
- [ ] 点击指数或市场摘要后，用户能够看到明确的下一层入口或反馈。

### 15.3 行业热点

- [ ] 行业列表至少展示行业名称、涨跌幅和领涨股。
- [ ] 行业列表默认按照涨跌幅降序排列。
- [ ] 用户可以切换到资金净流入排序，或看到该入口的原型反馈。
- [ ] 点击行业条目能够进入行业详情演示页或详情状态。
- [ ] 行业详情演示页包含行业表现、领涨股、资金方向和返回入口。

### 15.4 实时简报和快捷入口

- [ ] 首页展示 2-4 条具体简报，每条包含标题、类型或来源、时间。
- [ ] 点击简报能够进入详情状态或打开详情弹层，并能够返回首页。
- [ ] 首页提供自选、盯盘、涨跌排行、资金流向等快捷入口。
- [ ] 自选和盯盘入口能够进入当前项目已有业务路径。

### 15.5 状态与合规

- [ ] 页面能够演示加载、数据异常或非交易时段中的至少一种状态。
- [ ] 数据异常时保留必要的返回或刷新操作。
- [ ] 页面标记演示数据、更新时间和必要的风险提示。
- [ ] 不出现未经依据的个性化买卖建议。

### 15.6 视觉和原型质量

- [ ] 默认使用 390px 移动端亮色模式。
- [ ] 视觉优先参考 Figma Page 1，不使用通用金融卡片风。
- [ ] 通用图标使用当前项目图标资源或明确标注临时图标。
- [ ] 文本不重叠、卡片不越界、底部导航不遮挡主要内容。
- [ ] 页面中的主要点击路径可被产品经理在评审中直接演示。

## 16. 风险和待确认问题

### 16.1 待确认

- 行情首页首个正式市场是否只做 A 股，还是上线即支持港股和美股。
- 行业分类采用申万、证监会行业还是产品自定义分类。
- 资金流向数据使用实时、分钟级还是收盘后数据。
- 行情和行业数据的延迟标识及收费权限如何展示。
- 是否需要展示涨停/跌停、北向资金、龙虎榜等专业指标。
- 简报来源是人工编辑、第三方资讯还是 AI 摘要。
- 行业详情和指数详情是否属于本期同一需求，还是作为后续页面。

### 16.2 主要风险

- 行情数据时效不一致会造成指数、行业和简报之间的时间错位。
- 行业分类和资金口径不同会导致用户误解排行结果。
- 首屏模块过多会降低移动端扫描效率。
- 行业热点展示若与投资建议混淆，存在合规和误导风险。
- 原型使用演示数据时，评审人员可能误把内容当作真实行情，需要明确标注。

## 17. 建议的下一步

1. 产品确认行业分类、市场范围和数据口径。
2. 设计基于 Figma Page 1 确认行情首页的真实源稿结构和组件边界。
3. 使用 `stock-trading-design-system` 校准页面 token、组件和图标。
4. 使用 `stock-trading-prototype-design-system` 生成 390px HTML 原型，重点验证首页到行业详情、简报详情、自选和盯盘的基本信息架构。
5. 产品、设计、研发和测试共同走查首屏信息密度、交互路径、状态和数据说明。
