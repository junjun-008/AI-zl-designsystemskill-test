# 原型流程与交互规则

## 默认交付结构

每次生成 HTML 原型时，输出应包含：

1. 可运行的 HTML/CSS/JS。
2. 页面目的。
3. 模块列表。
4. 关键交互说明。
5. 数据假设。
6. 设计规则自检。

## 评审所需交互

根据需求选择必要交互，不做无意义动效：

- Tab 切换：用于行情分类、AI 交易状态、方案列表。
- 选中态：用于快捷入口、筛选项、方案卡片。
- 弹窗/抽屉：用于风险提示、确认操作、补充说明。
- 页面跳转：用于详情页、历史记录、相关股票。
- 加载/空/错误状态：当需求涉及数据异步或异常情况时必须补充。

## 页面修改流程

页面修改类需求按这个顺序处理：

1. 确认原页面模式。
2. 明确新增或调整的模块。
3. 保留原页面导航和主要结构。
4. 判断新增模块应该放在顶部、中部卡片流、列表内，还是底部操作区。
5. 只生成能解释需求的必要交互。

## 输出限制

- 不做复杂微交互。
- 不做最终视觉精修承诺。
- 不把 HTML 原型说成 Figma 组件库。
- 不把缺失图标伪装成真实图标。
