# 截图驱动改页流程

本文件用于已有页面修改、功能调整、模块新增。目标是让 Skill 保持轻量：不把每个页面写成固定模板，而是每次根据用户提供的 Figma 节点、截图或当前页面原型识别页面结构，再进行局部改造。

## 适用场景

- 用户提供 Figma 链接、截图、当前 HTML 原型或页面名称，并要求修改已有页面。
- 用户的需求来自 PRD、评审意见或功能调整说明。
- 页面视觉需要贴近当前 App，而不是重新设计。

## 不适用场景

- 从零设计全新业务形态。
- 生成营销页、海报、官网、桌面后台。
- 用户明确要求不参考截图或 Figma。

## 工作流程

### 1. 读取来源

优先级：

1. 用户提供的当前页面截图或 Figma 节点截图。
2. Figma `Page 1` 对应真实节点。
3. 当前本地 HTML 原型截图。
4. Skill 中的 token、组件规则和结构化资产。

如果用户要求“贴近某个页面”，但没有提供可读截图或节点，需要先说明缺口；能从 Figma 链接读取时，先读取再生成。

### 2. 识别当前页面结构

生成前必须先识别并保留以下结构：

- 状态栏和顶部导航。
- 页面主业务导航或标题区。
- 筛选、Tab、分组、快捷入口等控制区。
- 主内容区的结构类型：列表、卡片流、时间线、详情正文、表单。
- 固定操作区、底部导航、安全区。
- 当前页面的信息密度、圆角、分割线、字号节奏。

不要只根据 PRD 重画页面。

### 3. 标记需求影响区域

根据 PRD 或用户要求，把改动分成三类：

- `To Preserve`：必须原样保留的页面壳、导航、列表节奏。
- `To Modify`：需要替换或增强的现有区域。
- `To Add`：新增模块、状态、弹层或交互。

只有 `To Modify` 和 `To Add` 可以改变布局。没有被影响的区域必须保持原页面结构。

### 4. 局部改造

局部改造原则：

- 优先在原位置插入或增强，不重构整页。
- 新模块必须继承原页面的间距、密度、圆角、列表行高和导航位置。
- 新文案必须服务业务任务，不添加营销化解释。
- 如果只是评审原型，交互只覆盖理解需求所需状态。

### 5. 缺失素材处理

icon、股票 logo、插图、头像、复杂图表缺失时：

- 先查 `assets/icon-index.json` 或 Figma 节点。
- 找不到时使用低调占位。
- 输出说明中标注“图标待替换”或“素材待设计确认”。
- 不用随机图标、装饰 SVG、emoji 或无关图片替代。

### 6. 视觉自检

输出前必须检查：

- 页面主结构是否仍像来源截图。
- 顶部导航、业务 Tab、底部导航是否保留。
- 列表或卡片密度是否接近来源截图。
- 新增模块是否破坏原页面节奏。
- 是否出现通用金融 Dashboard、营销卡片、桌面说明区。
- 是否有遮挡、重叠、裁切、移动端安全区问题。

如生成 HTML 原型，优先用浏览器截图核对默认态和关键交互态。

## 输出要求

已有页面修改类输出应包含：

- 本地 HTML 原型文件。
- 使用的来源：截图、Figma 节点或本地原型。
- 修改点摘要。
- 交互说明。
- 假设和缺失素材说明。
- 简短自检结果。

不要在用户要求“只输出手机原型”时额外加入桌面说明区。
