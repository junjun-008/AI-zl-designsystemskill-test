# 案例库

案例用于解释规则来源。案例不是自动规则。

## CASE-001: Figma 文档区排版重叠

**背景：** 整理 Figma Components 页时，文档区新增审计摘要。

**问题：** 文档节点位置没有充分计算，导致命名表、暗色映射、审计摘要出现重叠。

**根因：** Figma 写入前没有计算目标区域尺寸，写入后没有立即做 overlap 校验。

**修正：** 后续任何 Figma 写入都必须先规划布局边界，写入后运行碰撞检测。

**规则影响：** 当前 MVP Skill 默认输出 HTML 原型，Figma 输出后置；未来 Figma 写入需要单独布局校验规则。

**来源：** 用户反馈和修复记录。

**后续 eval：** 当 Skill 支持 Figma 输出时，增加“新增文档或原型节点不得重叠”的检查。

## CASE-002: 自选页原型不像源稿

**背景：** 根据“自选页持仓分析与智能盯盘”PRD 生成 HTML 原型，用户要求贴近 Figma Page 1 的自选页。

**问题：** 生成结果虽然覆盖了持仓、盯盘、触发等功能，但视觉还原不足；底部导航、图标、列表密度、页面层次和真实自选页差距明显。

**根因：**

- Skill 还没有稳定的截图驱动改页流程，生成时容易从需求文档重画页面。
- icon 没有资产索引和缺失处理规则，导致占位风格不稳定。
- 股票列表、胶囊筛选、底部导航等跨页面组件规则不够细。
- 源稿自选页偏暗色，而用户要求亮色，亮色转译缺少逐项校准。

**修正：**

- 新增 `screenshot-driven-workflow.md`，要求已有页面修改必须先识别来源截图/Figma结构。
- 扩展 `components-shared.md`，补股票列表行、业务 Tab、胶囊筛选、状态标签、底部弹层等跨页面组件规则。
- 新增 `icon-index.json`，用语义索引约束 icon 来源和占位方式。
- eval 改为验证“PRD + 截图/Figma 改已有页面”，而不是验证某个固定页面模板。

**规则影响：** 不把 `watchlist-page` 等具体页面写死进 Skill。高频页面也通过截图/Figma 驱动识别结构，Skill 只沉淀通用组件、流程和自检。

**来源：** 用户对旧版自选页原型的反馈与后续确认。

**后续 eval：** 增加“自选页截图 + PRD”测试，检查是否保留源页面结构、列表密度和底部导航关系。
