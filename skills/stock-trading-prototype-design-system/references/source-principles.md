# 来源原则

## 核心原则

这个 Skill 的目标是生成符合当前 App 风格的原型，而不是泛化设计。

当模型不确定时，优先问：

> 这个规则或样式是否能在 Page 1、token、结构化资产或用户确认中找到来源？

## 来源等级

1. 用户提供的当前页面截图、Figma 节点截图或当前 HTML 原型截图。
2. Figma `Page 1` 真实页面。
3. `assets/design-tokens.css`。
4. `assets/page-patterns.json`、`assets/component-patterns.json`、`assets/source-map.json`。
5. 用户确认过的设计评审结论。
6. 明确标注的假设。

## 已有页面修改

已有页面修改不依赖固定页面模板。即使是高频页面，也应优先从用户提供的截图、Figma 节点或当前原型中识别结构。

必须先判断：

- 哪些区域属于当前页面壳，必须保留。
- 哪些区域是 PRD 影响范围，可以修改。
- 哪些区域是新增功能，需要插入。

不要因为 Skill 中存在相近页面模式，就跳过来源截图识别。

## Components 页状态

当前整理出来的 Components 页不能直接作为权威组件库。它可以作为参考，但进入正式规则前必须：

- 对应到 Page 1 的真实源节点。
- 明确组件边界。
- 补齐 token 和状态规则。
- 经过用户确认。

## 假设标注

如果需要补全未覆盖场景，输出中必须写：

- 哪一部分是源稿已有。
- 哪一部分是根据现有模式推断。
- 哪些素材需要设计师后续替换或确认。
